# Tool Adoption Records

Git-tracked snapshots of third-party Claude Code tools (MCP servers, skills, plugins, commands, hooks) that have been reviewed by InfoSec for adoption at FMFX.

Generated by the [`tool-security-analysis`](../skills/tool-security-analysis/SKILL.md) skill in **InfoSec mode**. Engineers requesting a tool produce a Jira ticket — InfoSec produces the record here.

## Layout

```
tool-adoption-records/
  <slug>/
    snapshot.json              ← canonical current state
    adoption-record.md         ← latest Confluence-paste markdown (reference copy)
    history/
      2026-05-06-snapshot.json ← prior snapshots, ordered by reviewed_on
```

Slugs are lowercase kebab-case and include a region/scope qualifier when relevant (e.g. `datadog-mcp-eu`, `figma-mcp`, `github-mcp-public`).

## How re-runs work

When `tool-security-analysis` runs in InfoSec mode against a slug that already has a `snapshot.json`:

1. The skill loads the prior snapshot.
2. It produces a fresh research pass.
3. It diffs the two semantically — surfaces tools added/removed, description changes, dependency churn, license/maintainer changes, new CVEs, transport/egress changes.
4. Each delta is tagged `[security-relevant]` or `[informational]`. The skill recommends a re-review before the new snapshot is committed.
5. The prior `snapshot.json` is moved to `history/<reviewed_on>-snapshot.json`.
6. The new `snapshot.json` and `adoption-record.md` are written.
7. The skill prints a suggested commit message; **the engineer commits manually** after reviewing the git diff.

## Schema

`snapshot.json` validates against [`../skills/tool-security-analysis/snapshot.schema.json`](../skills/tool-security-analysis/snapshot.schema.json). CI enforces this on every PR that touches `tool-adoption-records/**`.

## Sensitivity rules

This directory is committed to a public-shape repository. **Do not paste:**

- API keys, bearer tokens, or any credential values (record auth *types* and *scope names* only).
- Internal URLs whose existence is itself confidential.
- Customer identifiers, account IDs, or PII observed during research.
- Internal-only Confluence/Jira links that disclose product roadmap.

If a research step requires sensitive context, capture it in the InfoSec-internal Confluence page only — not in the snapshot or adoption record.

## CI

Changes under `tool-adoption-records/**` trigger the `Plugin version` workflow for snapshot schema validation **but do not require a plugin version bump**. See `.github/workflows/plugin-version.yml`.
