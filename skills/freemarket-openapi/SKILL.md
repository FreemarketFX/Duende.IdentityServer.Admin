# Freemarket OpenAPI Lookup

Single source of truth for Freemarket API specs is the workspace `openapi-specs` directory — no cached copy in this skill.

- File pattern: `{ApiName}Api_internal.json` (internal endpoints) or `{ApiName}Api_public.json` (public surface).
- Refresh: those files are regenerated by the design pipeline; if a spec is stale, ask the user to pull whichever repo hosts them. This skill does NOT fetch them.

## Resolve the spec dir

Walk up from cwd looking for an `openapi-specs/` directory. Stop at the first hit:

1. `./openapi-specs/`
2. `../openapi-specs/`
3. `../../openapi-specs/` (and keep walking up until you reach the filesystem root)
4. Sibling repos one level up: `../*/openapi-specs/` — `Glob pattern="*/openapi-specs/*Api_internal.json" path=".."` if step 3 misses.
5. Fallback: `C:\dev\freemarket\design\openapi-specs\` (workspace convention).

Use the first directory that contains `*Api_internal.json` files. If none found, tell the user and stop.

## How to use it

The user names an API and what they want to do — e.g. "ClientActions create beneficiary", "Platform users list", "MoneyMovement quote".

1. **Resolve the spec files.** Map the API name to the filename: `ClientActions` → `ClientActionsApi_internal.json` + `ClientActionsApi_public.json`, etc. If unsure, `Glob pattern="*Api_*.json" path="<spec-dir>"` to list candidates.

2. **Check both `_internal` and `_public` in parallel via subagents.** Spawn two `Explore` (or `general-purpose`) subagents in a single message — one per file — each with a tightly scoped prompt:

   > Search `<spec-dir>/ClientActionsApi_internal.json` for an endpoint matching "create beneficiary". Use Grep against `"/beneficiar`, `"post":`, `summary.*[Bb]eneficiary`, `tags.*Beneficiary`. When you find a candidate, Read a narrow window around the match (offset/limit) and extract: HTTP method, path, summary, request body `$ref`, response `$ref`s + status codes. If the body/response uses `$ref: "#/components/schemas/Foo"`, Grep `"Foo":` under `components/schemas` in the same file and include that schema. Report under 200 words: METHOD /path, summary, request shape, response shape per status.

   Run the `_public` agent with the same brief on the `_public.json` file. Parallel subagents keep the 100s-of-KB specs out of your context window.

3. **Reconcile the two results.** For UI prototypes prefer `_internal` (richer surface). Call out any divergence — e.g. internal has fields the public surface omits, or the endpoint only exists on one side.

4. **Quote the endpoint** to the user as `METHOD /path` with a one-line purpose, plus request/response shape the prototype should expect. Note which surface (internal vs public) it came from.

## Scope reminder

This skill surfaces **which endpoint to call** for HTML prototypes / mockups. It is not for generating server-side handlers, writing production wiring, or making live HTTP calls.
