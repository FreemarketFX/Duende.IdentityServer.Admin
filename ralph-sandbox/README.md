# Ralph Sandbox

An autonomous AI agent loop that runs Claude Code inside a Docker sandbox to implement features from a PRD (Product Requirements Document).

## Prerequisites

- **Docker Desktop** >= 4.62.0 (with Docker sandbox support)
- **Azure CLI** (`az`) — install from https://aka.ms/installazurecli
- **PowerShell** (pwsh)
- Claude credentials at `~/.claude/.credentials.json`
- **KSM CLI** (for local MCP secret resolution) — install from https://docs.keeper.io/en/secrets-manager/secrets-manager/secrets-manager-command-line-interface/init-command

## Usage

```powershell
# Run from the ralph-sandbox directory
./ralph-sandbox.ps1                              # Default: 10 iterations
./ralph-sandbox.ps1 -MaxIterations 20            # Custom iteration count
./ralph-sandbox.ps1 -LocalImage                  # Use local ralph-sandbox:latest (skip ACR pull)
./ralph-sandbox.ps1 -Prompt "what day is it?"    # DEBUG ONLY: Custom prompt (overrides default CLAUDE.md instruction)
```

| Parameter | Default | Description |
|-----------|---------|-------------|
| `-MaxIterations` | 10 | Maximum number of Claude iterations before stopping |
| `-SkipUpdateCheck` | off | Skip the `claude-shared` subtree freshness check |
| `-LocalImage` | off | Use the locally built `ralph-sandbox:latest` instead of pulling from ACR |
| `-Prompt` | _(CLAUDE.md)_ | Override the default prompt sent to Claude each iteration |
| `-LegacyRecreate` | off | Full `docker sandbox rm`+`create` between iterations (~5min/cycle). Default is in-place reset (~5s) plus per-call fresh `$HOME`. Use only if the in-place path regresses. |
| `-NoFreshHome` | off | Disable per-call fresh `$HOME`. Each `claude` exec normally runs with `HOME` pointed at a per-call `/tmp` dir mirrored from `/home/agent` minus transient `.claude/` state — this is what stops the second `claude` invocation deadlocking on a reused sandbox. Flip off only for differential testing. |
| `-DebugMode` | off | Attach the per-iteration sandbox-state snapshot and the 60s-silence forensic watchdog dump to the log. Adds noise; only useful when investigating a hang. |

The script expects to be located two levels below the target repo root (`<repo>/<parent>/ralph-sandbox/`). It derives `$RepoRoot` as `$ScriptDir.Parent.Parent`.

## How It Works

### Pre-flight Setup

1. Checks `claude-shared` subtree is up to date
2. Verifies PowerShell Core and Azure CLI auth (prompts `az login` if needed)
3. Pulls the latest sandbox image from ACR (unless `-LocalImage` is set)
4. Creates the working directory if it doesn't exist
5. Ensures working directory artifacts are in the repo root `.gitignore`
6. Syncs `settings.local.json` to the target repo's `.claude/` directory (with backup if changed)
7. Discovers MCP server manifest (`tasks/config/mcp-servers.yml`) and starts Docker-based MCP containers

### Sandbox Image (ACR)

The sandbox Docker image (`ralph-sandbox:latest`) is built by CI via GitHub Actions (`.github/workflows/build-sandbox-image.yml`) and pushed to Azure Container Registry at `fmfxukcontainerregistry.azurecr.io/ralph-sandbox:latest`.

On each run (unless `-LocalImage` is set), the script:
1. Checks you are logged into the correct Azure tenant (prompts `az login` if not)
2. Authenticates to ACR via `az acr login`
3. Compares local and remote image digests
4. Pulls the latest image only if it has changed (or no local image exists)

With `-LocalImage`, the ACR pull is skipped entirely and the local `ralph-sandbox:latest` is used. Build a local image with:

```powershell
ralph-sandbox/docker/build-local.ps1
```

### Sandbox Management

- Names the sandbox `claude-<repo-folder-name>`
- Creates the sandbox if it doesn't exist
- Starts the sandbox if it's stopped
- Syncs `config/proxy-config.yml` to `~/.sandboxd/proxy-config.json` **before** sandbox creation, so the sandbox inherits a deny-by-default network policy from the moment it starts (allowed domains are listed in the YAML file)

### Secret Management (KSM)

Secrets (NuGet feed PAT, Datadog API keys) are managed via [Keeper Secrets Manager](https://docs.keeper.io/en/secrets-manager/) (KSM). The template file `config/sandbox-persistent.template` contains `keeper://` URIs that are resolved at runtime.

**Sandbox flow:**
1. On sandbox creation, `ralph-sandbox.ps1` fetches the KSM config from Azure Key Vault and writes it to `~/.claude/.keeper`
2. The sandbox entrypoint (`entrypoint.sh`) imports the KSM profile and runs `ksm interpolate` on the template
3. Resolved secrets are written to `/etc/sandbox-persistent.sh`, which is auto-sourced by the sandbox shell
4. Before each iteration, `envsubst` resolves `${...}` references in `.mcp.json` using the sandbox env vars

**Local flow (via `mcp-startup.ps1`):**
1. Detects `${...}` references in the generated `.mcp.json`
2. Fetches the KSM profile from Azure Key Vault
3. Runs `ksm interpolate` on the template to resolve secrets
4. Substitutes resolved values into `.mcp.json`
5. Cleans up the temporary KSM profile

### MCP Servers

MCP (Model Context Protocol) servers give Claude access to external tools (browser automation, observability, etc.). Server configuration lives in `tasks/config/mcp-servers.yml` in the target repo.

**Supported servers:**

| Server | Type | Description |
|--------|------|-------------|
| Playwright | Docker (local container) | Browser automation via headless Chromium — see [`mcp/playwright/README.md`](mcp/playwright/README.md) |
| Datadog | HTTP (remote) | Observability and log querying — see [`mcp/datadog/README.md`](mcp/datadog/README.md) |

**In the sandbox:** The entrypoint reads the MCP manifest, starts Docker-based MCP containers, and generates `.mcp.json`. On iteration 1, a startup gate prompt (`mcp/mcp-startup-prompt.md`) is prepended to verify containers are healthy and MCP tools are available before proceeding.

**For local development (outside the sandbox):**

```powershell
# Start MCP servers and generate .mcp.json
claude-shared/ralph-sandbox/mcp/mcp-startup.ps1

# Stop servers and remove .mcp.json
claude-shared/ralph-sandbox/mcp/mcp-startup.ps1 -Down
```

See the per-server READMEs linked above for detailed setup, domain restrictions, and troubleshooting.

### Iteration Loop

Each iteration:
1. Resolves `${...}` env var references in `.mcp.json` using sandbox env vars (from KSM)
2. Prepends the MCP startup gate prompt on iteration 1 (verifies containers are healthy and tools are connected)
3. Invokes Claude inside the sandbox with `--permission-mode acceptEdits`
4. Streams output as JSON, parsing tool calls and results to stdout
5. Logs full JSON to `ralph-sandbox.log`
6. Checks for `<promise>COMPLETE</promise>` in the result to stop early
7. Exits with code 0 on completion, or 1 if max iterations reached

### Run Archival

When the `branchName` in `prd.json` changes between runs, the previous `prd.json` and `progress.txt` are archived to `tasks/archive/<date>-<branch>/`.

## Teardown

```powershell
# Remove the sandbox for the current repo only (prompts for confirmation)
./sandbox-teardown.ps1

# Remove ALL sandboxes and sandbox images across all repositories
docker sandbox reset
```

## Network Policy

The sandbox network policy is defined in `config/proxy-config.yml` and synced to `~/.sandboxd/proxy-config.json` before sandbox creation. The policy is deny-by-default.

**Why this approach?** Docker Desktop ships a default `proxy-config.json` with `"default": "allow"` and pre-populated entries for OpenAI, Google APIs, and other services we don't use. The previous approach — applying policy via `docker sandbox network proxy` after creation — left a window where the sandbox ran under the permissive default, and the default config could be tampered with (by the agent or other processes) to re-add unwanted domains. By writing our own `proxy-config.json` *before* `docker sandbox create`, the sandbox inherits the locked-down policy from the moment it starts, with no gap and no reliance on Docker Desktop's defaults.

To allow a new domain, add it to `network.allowedDomains` in `config/proxy-config.yml`. For hosts that require certificate pinning (reject the sandbox proxy's CA), add them to `network.bypassDomains` instead — bypass traffic goes direct without TLS interception.

## Claude Permissions

`settings.local.json` defines what Claude can and cannot do inside the sandbox:

- **Allowed**: `Bash`, `Write`, `Edit`, `Read`, `Glob`, `Grep`
- **Denied**: Network tools (`curl`, `wget`, `nc`, `ssh`, etc.), inline script execution (`python -c`, `node -e`), and Azure CLI (`az`)


## Troubleshooting

| Error | Cause | Resolution |
|-------|-------|------------|
| `failed to load template image: no Docker context found for sandbox {name}: docker context not found` | A previous sandbox removal or reset failed to remove the associated docker context folder | Remove all empty folders beneath `~/.docker/contexts/meta/` and try again |
